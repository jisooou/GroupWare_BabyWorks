## 개발 포인트 

### 1. 수신자 기준 읽음 처리

### 문제 상황

쪽지 읽음 기능 구현 시, 수신자와 발신자 중 한 명만 쪽지를 열어도 읽음 처리되는 문제가 발생했다.

이로 인해 실제로는 수신자가 확인하지 않았음에도 읽음 쪽지함으로 이동하는 비정상적인 흐름이 발생했다.

### 해결 방법

읽음 처리 조건을 단순히 “쪽지를 열었는지 여부”가 아니라,

**현재 로그인한 사용자가 해당 쪽지의 수신자인지 확인한 경우에만 읽음 처리**하도록 로직을 변경했다.

### 핵심 로직

- 현재 로그인한 사용자의 사원번호를 조회
- 해당 쪽지의 수신자 정보와 비교
- 두 값이 일치할 경우에만 읽음 상태로 변경
- 발신자는 쪽지를 열어도 읽음 처리되지 않도록 분리

### 2. JavaScript를 활용한 현재 시간 표시

### 문제 상황

화면에 현재 시간을 표시해야 했지만, 별도의 라이브러리 없이 구현해야 했고,

시간이 정적으로 표시되면 실제 시간과 맞지 않는 문제가 있었다.

### 해결 방법

JavaScript의 Date 객체를 활용하여 현재 시간을 계산하고,

**1초마다 갱신되도록 하여 실시간으로 변화하는 시계를 구현**했다.

### 핵심 로직

- 페이지 로드 시 현재 시간 계산
- 시/분/초를 두 자리 문자열로 변환하여 포맷 유지
- setInterval을 활용해 1초마다 시간 업데이트
- DOM 요소에 현재 시간을 지속적으로 반영

### 3. 보낸 사람 이름 기준 검색 기능

### 문제 상황

쪽지 검색 기능 구현 시, 검색 기준이 명확하지 않아

현재 로그인한 사용자(수신자) 기준으로 검색되는 문제가 있었다.

하지만 실제 요구사항은 **보낸 사람 이름 기준으로 검색**하는 것이었다.

### 해결 방법

검색 조건을 수정하여,

**현재 로그인한 사용자가 받은 쪽지 중에서 보낸 사람 이름을 기준으로 검색**되도록 변경했다.

### 핵심 로직

- 검색어 입력 → 서버로 전달
- 현재 로그인한 사용자 기준으로 쪽지 범위 제한
- 보낸 사람 이름 컬럼을 기준으로 LIKE 검색 수행
- 결과를 최신순으로 정렬하여 반환

### 4. 개별 선택 및 전체 선택 기반 삭제 기능

### 문제 상황

쪽지를 하나씩 삭제해야 하는 구조는 사용자 경험이 좋지 않았고,

여러 개의 쪽지를 한 번에 처리할 수 있는 기능이 필요했다.

### 해결 방법

**개별 선택 체크박스 + 전체 선택 체크박스**를 활용하여

여러 개의 쪽지를 한 번에 선택하고 삭제할 수 있도록 구현했다.

### 핵심 로직

- 전체 선택 체크 시 모든 항목 체크 상태 동기화
- 선택된 쪽지들의 ID를 배열 형태로 수집
- 서버로 리스트 형태로 전달
- 백엔드에서 일괄 처리 (휴지통 이동 또는 삭제)

### 5. WebSocket 기반 실시간 쪽지 알림

### 문제 상황

기존 방식에서는 사용자가 페이지를 새로고침해야만 새로운 쪽지를 확인할 수 있었고,

즉각적인 알림이 제공되지 않아 사용자 경험이 떨어지는 문제가 있었다.

### 해결 방법

WebSocket을 활용하여 서버와 클라이언트 간 연결을 유지하고,

**쪽지 작성 시 수신자에게 실시간으로 알림을 전달**하도록 구현했다.

또한 알림을 DB에 저장하여 단순 실시간 알림이 아닌

**미확인 알림 조회 및 읽음 처리까지 가능한 구조로 확장**했다.

### 핵심 로직

- WebSocket 연결을 통해 클라이언트와 서버 연결 유지
- 쪽지 작성 시 알림 메시지 생성
- 알림 데이터를 DB에 저장
- WebSocket을 통해 모든 연결된 세션에 알림 전송
- 페이지 진입 시 미확인 알림 조회
- 알림 클릭 시 읽음 처리
